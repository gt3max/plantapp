# PlantApp — 02. Smart Identify: План реализации

## Обзор

Идентификатор растений — лицо приложения и сайта. Должен работать точно, надёжно и бесплатно для пользователя. Принцип суперпозиции: несколько источников дополняют друг друга, при падении одного — остальные подхватывают.

**Текущее состояние:** PlantNet only (один источник). SerpAPI и Perenual подключены но логика примитивная (SerpAPI только при падении PlantNet).

**Целевое состояние:** 2-3 источника параллельно, merge результатов, graceful degradation, единый endpoint для приложения и сайта.

## 2. Умная логика (Smart Identify)

### 2.1 Основной flow

```
Фото пользователя
       ↓
┌─────────────────────────────────────────────┐
│  ПАРАЛЛЕЛЬНЫЙ ЗАПУСК (ThreadPoolExecutor)   │
│                                              │
│  Thread 1: PlantNet (если лимит не исчерпан)│
│  Thread 2: iNaturalist (всегда)             │
│                                              │
│  Timeout: 30 секунд каждый                  │
└──────────────┬──────────────────────────────┘
               ↓
┌─────────────────────────────────────────────┐
│  MERGE РЕЗУЛЬТАТОВ                           │
│                                              │
│  Case 1: Оба ответили, совпали              │
│    → Высокая уверенность (consensus)        │
│    → score = max(scoreA, scoreB)             │
│                                              │
│  Case 2: Оба ответили, расходятся           │
│    → Берём тот у кого score выше             │
│    → Если разница < 20% — показываем оба    │
│    → Если score < 50% → вызываем SerpAPI    │
│                                              │
│  Case 3: Один ответил, другой нет           │
│    → Используем того кто ответил             │
│                                              │
│  Case 4: Никто не ответил                   │
│    → SerpAPI Lens (fallback)                 │
│    → Если и он не ответил → ошибка           │
└──────────────┬──────────────────────────────┘
               ↓
┌─────────────────────────────────────────────┐
│  ПАРАЛЛЕЛЬНОЕ ОБОГАЩЕНИЕ                     │
│  (пока merge происходит, уже запускаем)     │
│                                              │
│  Thread A: Turso DB lookup (по scientific)   │
│  Thread B: Trefle (таксономия)              │
│  Thread C: Wikipedia (описание)             │
└──────────────┬──────────────────────────────┘
               ↓
         Финальный ответ пользователю
```

### 2.2 Экономия SerpAPI

SerpAPI (250/мес) вызывается ТОЛЬКО когда:
1. Оба основных источника дали score < 50% (спорный результат)
2. Оба основных источника разошлись во мнении
3. Оба основных источника не ответили (fallback)

При нормальной работе PlantNet + iNaturalist — SerpAPI не тратится.

**Расчёт:** ~5-10% запросов пойдут в SerpAPI. При 450 запросах/день (1000 пользователей) → ~30-45 SerpAPI запросов/день → ~900-1350/мес. Превышает 250. Решение: при исчерпании SerpAPI — работаем без него, PlantNet + iNaturalist достаточно.

### 2.3 Rate limiting (пользователи)

| Тип | Identify/день | Обоснование |
|-----|---------------|-------------|
| Анонимный (без регистрации) | 1 | Попробовать, не тратить ресурсы |
| Бесплатный (зарегистрирован) | 3 | Достаточно для хобби-использования |
| Premium | 20 | Активный пользователь |
| Admin | Безлимит | Разработка и тестирование |

### 2.4 Rate limiting (сервисы)

```python
# Глобальные счётчики (DynamoDB)
plantnet_daily_count    # Текущий расход PlantNet (лимит: 500)
serpapi_monthly_count   # Текущий расход SerpAPI (лимит: 250)

# Логика
if plantnet_daily_count >= 480:    # 96% — оставляем запас
    skip_plantnet = True           # iNaturalist справится один

if serpapi_monthly_count >= 240:   # 96% — оставляем запас
    skip_serpapi = True            # только PlantNet + iNaturalist
```

## 4. Безопасность

### 4.1 Защита API

- **JWT авторизация** — все identify запросы требуют валидный токен (кроме анонимных по IP)
- **Rate limiting** — DynamoDB счётчики с TTL, per-user и per-IP
- **Image validation** — проверка magic bytes (JPEG/PNG/WebP), отклонение невалидных файлов
- **Size limit** — максимум 10 MB на изображение
- **Timeout** — 30 сек на каждый сервис, 45 сек общий
- **No PII logging** — не логируем изображения, только scientific names и scores

### 4.2 Защита ключей

- API ключи в Lambda environment variables (не в коде)
- SECRETS.txt — единственный источник правды (в .gitignore)
- Ключи НЕ передаются клиенту — все вызовы server-side (Lambda)
- При компрометации ключа — перегенерация + update Lambda env

### 4.3 Abuse protection

- Анонимные пользователи: 1/день по IP (защита от ботов)
- Burst protection: максимум 5 запросов в минуту с одного user_id
- Подозрительная активность: если один IP создаёт много аккаунтов — блокировка

## 5. Проблемы при масштабировании (1000+ пользователей)

### 5.1 PlantNet лимит (500/день)

**Проблема:** При 1000 пользователей (150 активных/день, 450 запросов) — PlantNet на пределе.

**Решение:**
1. Кэширование: одно и то же растение (по хэшу фото) — не запрашивать повторно
2. iNaturalist как primary при исчерпании PlantNet лимита
3. Умное распределение: утром PlantNet + iNaturalist, ближе к лимиту — только iNaturalist
4. При росте до 2000+ — платный PlantNet ($50/мес = 5,000/день)

### 5.2 SerpAPI лимит (250/мес)

**Проблема:** При 5-10% спорных запросов от 1000 пользователей — 250/мес не хватит.

**Решение:**
1. SerpAPI только для low confidence (< 50%) — это ~2-3% запросов
2. Глобальный счётчик — при приближении к лимиту, отключаем
3. Без SerpAPI точность всё равно 85%+ (PlantNet + iNaturalist)
4. При росте — платный SerpAPI ($75/мес = 5,000/мес)

### 5.3 Lambda cold start

**Проблема:** Первый запрос после простоя = 3-5 сек дополнительно (Lambda загружается).

**Решение:**
1. Provisioned concurrency ($) — не нужно на MVP
2. Warmup ping каждые 5 минут (CloudWatch Events) — бесплатно
3. Клиент показывает "Identifying..." с анимацией (пользователь ждёт 2-5 сек нормально)

### 5.4 Параллельность в Lambda

**Проблема:** ThreadPoolExecutor в Lambda ограничен. Много параллельных вызовов к внешним API = таймауты.

**Решение:**
1. Максимум 3 потока (PlantNet + iNaturalist + enrichment)
2. Timeout 30 сек на каждый
3. Если один поток завис — не ждём, отдаём результат от остальных

### 5.5 Кэширование (ключевая оптимизация)

**Проблема:** Одно и то же растение фотографируют тысячи раз. Jade Plant, Monstera, Orchid — топ-20 растений это 60% всех запросов.

**Решение:**
```
1. При успешном identify → сохраняем результат в DynamoDB
   Key: SHA256(научное_имя + семейство)
   TTL: 30 дней
   
2. При новом identify → после получения научного имени
   → проверяем кэш
   → если есть → добавляем enrichment из кэша (0 запросов к Trefle/Wikipedia)
   
3. Это НЕ кэш фото (фото всегда уникальны)
   Это кэш данных об известных растениях
```

**Эффект:** При 1000 пользователях, 60% запросов на топ-20 растений → enrichment почти бесплатный.

### 5.6 Веб-сайт (plantapp.pro)

**Проблема:** Сайт доступен всем. Нет регистрации. Потенциально неограниченный трафик.

**Решение:**
1. Единый endpoint `/plants/identify` — сайт и приложение используют один и тот же
2. Сайт = анонимный пользователь → 1 identify/день по IP
3. При регистрации на сайте → 3/день (как в приложении)
4. Burst protection: 5 req/мин с одного IP
5. CORS: разрешить только plantapp.pro и приложение

### 5.7 Регионы

**Проблема:** PlantNet и SerpAPI могут быть медленными или недоступными в некоторых регионах (Китай, Россия).

**Решение:**
1. Lambda в eu-central-1 (Франкфурт) — хорошая связность с Европой и Ближним Востоком
2. iNaturalist (US-based) — хорошая связность с Америкой и Азией
3. Если один сервис таймаутится (30 сек) — другой уже ответил
4. При росте в конкретном регионе — добавить Lambda в us-east-1 или ap-southeast-1

## 6. Совместимость с сайтом (polivalka-web)

### 6.1 Текущее состояние сайта

Файл: `polivalka-web/identify.html`
- Использует тот же endpoint: `POST /plants/identify`
- Сжатие фото (1024px, JPEG 70%) на клиенте
- Показывает top 5 результатов
- Кнопка "Save" → `POST /plants/save`

### 6.2 Что изменится

**Ничего на клиенте.** Вся логика внутри Lambda. Сайт отправляет тот же запрос, получает расширенный ответ.

Новые поля в ответе (backwards compatible):
- `confidence` — сайт может показать badge "High confidence ✓"
- `consensus` — "Confirmed by 2 sources"
- `sources_agreed` — список подтвердивших сервисов
- `enrichment` — дополнительные данные (сайт может показать описание)
- `identify_remaining` — сколько осталось определений сегодня

Старые поля (`results`, `source`, `score`) — без изменений.

### 6.3 Обновление сайта

После обновления Lambda — сайт автоматически получает улучшенный identify. Можно обновить UI сайта чтобы показывать confidence и consensus, но не обязательно для MVP.

## 7. Порядок реализации

### Шаг 1: iNaturalist интеграция в Lambda
- Написать функцию `identify_with_inaturalist(image_bytes)`
- Формат: POST к iNaturalist vision API
- Парсинг ответа в наш формат
- Тесты: 5 разных растений

### Шаг 2: Merge логика
- Переписать `identify_plant_handler()` 
- PlantNet + iNaturalist параллельно (ThreadPoolExecutor)
- Merge результатов: consensus detection, score comparison
- Confidence level calculation

### Шаг 3: Smart SerpAPI
- SerpAPI вызывается только при:
  - Оба основных < 50% score
  - Результаты расходятся
  - Оба основных не ответили
- Глобальный счётчик расхода SerpAPI (DynamoDB)

### Шаг 4: Service rate limiting
- Глобальные счётчики PlantNet (500/день) и SerpAPI (250/мес)
- Graceful degradation: при исчерпании — пропускаем сервис
- Логирование расхода

### Шаг 5: Enrichment pipeline
- Trefle интеграция (таксономия, характеристики)
- Turso DB lookup (полная карточка если есть)
- Wikipedia summary (описание)
- Всё параллельно

### Шаг 6: Кэширование enrichment
- DynamoDB таблица для кэша (научное имя → данные)
- TTL 30 дней
- Значительная экономия при повторных запросах

### Шаг 7: Безопасность
- Burst protection (5 req/мин)
- Image size validation (10 MB max)
- Анонимные лимиты: 1/день по IP
- CORS whitelist

### Шаг 8: Тестирование
- 20 разных растений (популярные + редкие)
- Проверка fallback (отключить PlantNet → iNaturalist справляется?)
- Проверка consensus (оба согласны? расходятся?)
- Нагрузочный тест: 50 запросов подряд
- Проверка с сайта (polivalka-web/identify.html)

### Шаг 9: Deploy + мониторинг
- Deploy Lambda
- CloudWatch алерты: error rate > 5%, latency > 10s
- Dashboard: запросы/день, расход PlantNet, расход SerpAPI
- Warmup ping каждые 5 минут
